# Alerting Platform PoC

A self-contained alerting platform proof-of-concept that runs entirely in Docker Compose. It ingests real-time events from Kafka into ClickHouse, evaluates configurable CEL-based alert rules, surfaces firing alerts in a React UI, and uses AWS Bedrock (Amazon Nova Micro) to generate rule expressions from plain-English descriptions.

---

## Prerequisites

- **Docker** 24+ and **Docker Compose** v2.20+
- An **AWS account** with Amazon Bedrock access in `us-east-1` (or your chosen region), with the `amazon.nova-micro-v1:0` model enabled
- A **Git repository** (e.g. GitHub) you control, which will host the Spring Cloud Config configuration file

---

## Setup

### 1. Copy and fill in the environment file

```bash
cp .env.example .env
```

Open `.env` and fill in your values:

| Variable | Description |
|---|---|
| `AWS_ACCESS_KEY_ID` | AWS access key with Bedrock permissions |
| `AWS_SECRET_ACCESS_KEY` | Corresponding secret key |
| `AWS_DEFAULT_REGION` | AWS region where Bedrock is enabled (e.g. `us-east-1`) |
| `CONFIG_GIT_URI` | HTTPS URL of a Git repo you control (e.g. `https://github.com/your-org/alerting-config`) |
| `CONFIG_GIT_TOKEN` | Personal Access Token (PAT) with `repo` read scope on that repo |

### 2. Push the config file to your Git repo

The Spring Cloud Config Server reads `alerting-service.yml` from the root of the branch `main` in `CONFIG_GIT_URI`. Push the provided file:

```bash
# Clone your config repo (or create a new one)
git clone https://github.com/your-org/alerting-config
cd alerting-config

# Copy the config file from this PoC directory
cp /path/to/poc/alerting-config/alerting-service.yml .

# Commit and push
git add alerting-service.yml
git commit -m "Add alerting-service config"
git push origin main
```

The `alerting-service.yml` file declares two Kafka event sources (`player_login` and `player_deposit`) that the backend subscribes to at startup.

## Run

From the `poc/` directory:

```bash
docker-compose up --build
```

The first build downloads Maven/NPM dependencies and compiles everything — this takes 3–5 minutes. Subsequent starts reuse cached Docker layers.

Services start in dependency order. The `backend` waits for PostgreSQL, Redis, ClickHouse, Kafka, and the Config Server to all pass their health checks before it starts.

## Pre-seeded Rules

Two rules are inserted automatically by the Flyway migration (`V1__init.sql`) when the database is first created:

### 1. Large Deposit Alert (HIGH severity)

**Source:** `player_deposit`
**Trigger event type:** `deposit`
**CEL expression:** `payload.amount > 500.0`
**Behaviour:** Fires whenever a single deposit event carries an `amount` field greater than 500. Approximately 30% of deposits generated by the event generator exceed this threshold, so alerts appear within seconds of startup.
**Resolution mode:** MANUAL — must be acknowledged or resolved by a human in the UI.

### 2. Suspicious Login Frequency (MEDIUM severity)

**Source:** `player_login`
**Trigger event type:** `login`
**CEL expression:** `agg_count("player_login", 1) > 3`
**Behaviour:** Fires when a single player has more than 3 login events in the last day (window = 1 day), as counted by a live ClickHouse aggregation query. The event generator sends a "login burst" of 5 rapid logins for a random player every 30 seconds, which reliably triggers this rule.
**Resolution mode:** AUTO — the backend re-evaluates the expression every 5 minutes; if the count drops to 3 or below (window slides), the alert auto-resolves.

## How the Demo Works

```
event-generator
  → Kafka topics: player.login, player.deposit
      ↓
ClickHouse Kafka Table Engine
  → consumes topics into events_player_login, events_player_deposit
  → MaterializedViews build daily/monthly rollup aggregates
      ↓
Java Backend (alerting-service)
  → KafkaConsumerManager subscribes to both topics via Spring Kafka
  → RuleEvaluationOrchestrator evaluates all enabled rules for each event
  → CelEvaluator resolves agg_* function calls by querying ClickHouse
  → AlertManager deduplicates and persists new Alert records to PostgreSQL
      ↓
Frontend (React + Ant Design)
  → Polls GET /api/v1/alerts every 5 seconds — alerts appear in real time
  → Event Feed page polls GET /api/v1/events every 2 seconds (from ClickHouse)
  → Rule Management page: create, enable/disable, delete rules
  → AI Generation: describe a rule in plain English → AWS Bedrock generates CEL
```

### Creating a new rule with AI

1. Go to **Rules** in the left nav.
2. Click **New Rule**.
3. Select an Event Source.
4. In the "AI-Assisted CEL Generation" section, type a plain-English description, e.g.: `Alert when a player makes more than 5 deposits in the last 7 days`.
5. Click **Generate** — the backend calls AWS Bedrock (Amazon Nova Micro) and populates the CEL expression and summary fields.
6. Review the generated expression, adjust if needed, and click **Create Rule**.

## Stopping

To stop all containers and remove volumes (full reset including database and Kafka data):

```bash
docker-compose down -v
```

To stop without removing data (containers restart from where they left off):

```bash
docker-compose down
```
